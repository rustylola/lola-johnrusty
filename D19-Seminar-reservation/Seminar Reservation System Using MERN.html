Seminar Reservation System Using MERN Stack

Project Description

The Seminar Reservation System Using MERN Stack is a full-stack web application designed to streamline the process of organizing, managing, and booking seminars. The system provides an intuitive user interface for both attendees and administrators, ensuring an efficient workflow for seminar registration, booking management, and payment processing. Built using the MERN (MongoDB, Express.js, React.js, Node.js) stack, the system leverages modern web development technologies to deliver a responsive, scalable, and secure platform.

Features

User Features

  1. User Registration and Login:

    • Secure registration with input validation for personal details such as first name, last name, email, and password.
    • Encrypted password storage using bcrypt.
    • Authentication using JSON Web Tokens (JWT).

  2. User Profile Management:

    • View and update personal profile details.
    • Secure password update functionality.

  3. Seminar Browsing:

    • View a list of upcoming seminars with details such as title, description, date, time, venue, speaker information, fee, and available slots.
    • Search and filter seminars by title, date, or speaker.

  4. Booking System:

    • Book a seminar by selecting a seminar from the list and providing necessary payment details.
    • Upload proof of payment (e.g., receipt or bank deposit slip).
    • View a list of all bookings, including booking status (pending, confirmed).

  5. Payment Processing:

    • Generate a payment intent via Stripe integration for online payment.
    • Option for offline payment with proof submission.

  6. Notification System:

    • Receive email notifications for booking confirmations or changes.
    • Receive reminders about upcoming seminars.

Admin Features

  1. Admin Authentication:

    • Admin login with elevated privileges to manage the platform.

  2. Seminar Management:

    • Add new seminars with details such as title, description, date, time, venue, speaker information, fee, and available slots.
    • Update seminar details (e.g., time, venue, or fee).
    • Delete seminars that are no longer available.

  3. User Management:

    • View a list of registered users.
    • Manage user accounts (e.g., deactivate, update roles).

  4. Booking Management:

    • View and manage seminar bookings.
    • Approve or reject bookings based on proof of payment.
    • Update booking statuses (e.g., pending, confirmed).

  5. Reporting:

    • View analytics for seminars, such as the number of attendees, revenue generated, and seminar popularity.
    • Export data for reporting purposes.

  6. Payment Reconciliation:

    • Validate proof of payment submitted by users for offline payments.
    • Monitor Stripe payment logs for online transactions.

  7. Notifications:

    • Send bulk emails to users for seminar updates or promotions.

Workflow

User Workflow

  1. Registration and Login:

    • User creates an account by providing necessary details.
    • Logs in to access the dashboard.

  2. Browsing and Booking:

    • The user browses the seminar list and selects an event.
    • The user provides payment details for booking.
    • If offline payment is selected, the user uploads proof of payment.

  3. Booking Confirmation:

    • The system confirms booking based on availability.
    • The user receives an email notification about the booking status.

  4. Attending Seminars:

    • Users attend the seminar on the scheduled date and time.
    • Receive follow-up materials or certificates after attendance.

Admin Workflow

  1. Login:

    • Admin logs in with elevated privileges.

  2. Seminar Creation:

    • Admin creates a seminar by filling out a form with the required details.
    • Newly created seminars are displayed to users.

  3. User and Booking Management:

    • Admin views user bookings and validates payment proofs.
    • Updates booking statuses and sends notifications as necessary.

  4. Seminar Updates:

    • Admin updates seminar details or cancels events if required.
    • Notifies registered users about changes.

  5. Reporting and Analysis:

    • Admin views detailed reports for seminar participation, revenue, and user activity.
    • Uses insights to improve seminar offerings.

System Benefits

  • Users:

    • Seamless registration and booking experience.
    • Transparency in seminar details and availability.
    • Multiple payment options for convenience.
    • Personalized notifications and updates.

  • Admins:

    • Efficient seminar and user management.
    • Accurate tracking of bookings and payments.
    • Automated notifications to reduce manual effort.
    • Analytical insights for strategic decision-making.

Step 7: Controllers

controllers/authController.js

import bcrypt from 'bcryptjs';
import jwt from 'jsonwebtoken';
import User from '../models/User.js';

const register = async (req, res) => {
  try {
    const { firstName, lastName, email, password } = req.body;

    const existingUser = await User.findOne({ email });
    if (existingUser) return res.status(400).json({ error: 'User already exists' });

    const hashedPassword = await bcrypt.hash(password, 10);
    const user = await User.create({ firstName, lastName, email, password: hashedPassword });

    res.status(201).json({ message: 'User registered successfully', user });
  } catch (error) {
    res.status(500).json({ error: 'Server error' });
  }
};

const login = async (req, res) => {
  try {
    const { email, password } = req.body;

    const user = await User.findOne({ email });
    if (!user) return res.status(404).json({ error: 'User not found' });

    const isMatch = await bcrypt.compare(password, user.password);
    if (!isMatch) return res.status(400).json({ error: 'Invalid credentials' });

    const token = jwt.sign({ id: user._id }, process.env.JWT_SECRET, { expiresIn: '1h' });
    res.status(200).json({ token, user: { id: user._id, name: `${user.firstName} ${user.lastName}`, email: user.email, role: user.role  } });
  } catch (error) {
    res.status(500).json({ error: 'Server error' });
  }
};

export { register, login };

controllers/bookingController.js

import Booking from '../models/Booking.js';
import Seminar from '../models/Seminar.js';

const createBooking = async (req, res) => {
  try {
    const { seminarId } = req.body;

    const seminar = await Seminar.findById(seminarId);
    if (!seminar) return res.status(404).json({ message: 'Seminar not found' });
    if (seminar.slotsAvailable <= 0) return res.status(400).json({ message: 'Seminar is full' });

    const booking = await Booking.create({ user: req.user.id, seminar: seminarId });
    seminar.slotsAvailable -= 1;
    await seminar.save();

    res.status(201).json({ message: 'Booking created successfully', booking });
  } catch (error) {
    res.status(500).json({ message: 'Error creating booking', error });
  }
};

const getUserBookings = async (req, res) => {
  try {
    const bookings = await Booking.find({ user: req.user.id }).populate('seminar');
    res.status(200).json(bookings);
  } catch (error) {
    res.status(500).json({ message: 'Error fetching bookings', error });
  }
};

const updateBookingStatus = async (req, res) => {
  try {
    const { id } = req.params;
    const { paymentStatus } = req.body;

    // Validate paymentStatus
    if (!['pending', 'confirmed', 'rejected'].includes(paymentStatus)) {
      return res.status(400).json({ message: 'Invalid payment status' });
    }

    // Find and update booking
    const updatedBooking = await Booking.findByIdAndUpdate(
      id,
      { paymentStatus },
      { new: true, runValidators: true }
    );

    if (!updatedBooking) {
      return res.status(404).json({ message: 'Booking not found' });
    }

    res.status(200).json({
      message: 'Booking status updated successfully',
      booking: updatedBooking,
    });
  } catch (error) {
    res.status(500).json({ message: 'Error updating booking status', error });
  }
};

export { createBooking, getUserBookings, updateBookingStatus };

Step 9: Middleware

middleware/authMiddleware.js

import jwt from 'jsonwebtoken';
import User from '../models/User.js';

const authMiddleware = async (req, res, next) => {
    const token = req.header('Authorization')?.split(' ')[1]; // Extract token after "Bearer"

  if (!token) return res.status(401).json({ message: 'No token, authorization denied' });

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = await User.findById(decoded.id).select('-password');
    next();
  } catch (error) {
    res.status(401).json({ message: 'Token is not valid' });
  }
};

export default authMiddleware;

middleware/adminMiddleware.js

const adminMiddleware = (req, res, next) => {
  if (req.user && req.user.role === 'admin') {
    next();
  } else {
    res.status(403).json({ message: 'Access denied: Admins only' });
  }
};

export default adminMiddleware;

5. Update Booking Status

Endpoint: PUT /api/bookings/:id
Description: Retrieves the logged-in user's bookings.

Request:

  • Method: PUT
  • URL: http://localhost:5000/api/bookings/:id (Replace :id with the actual booking ID.)
  • Headers:
    • Authorization: Bearer <ADMIN_JWT_TOKEN>
  • Body (JSON):
{
  "paymentStatus": "confirmed"
}

Response:

  • Success (200):
{
  "message": "Booking status updated successfully",
  "booking": {
    "_id": "64d0e456b8a6b154f7d2a8a3",
    "seminar": "64d0e123b8a6b154f7d2a8a2",
    "user": "64d0e0f2b8a6b154f7d2a8a4",
    "paymentStatus": "confirmed"
  }
}
  • Error (404):
{
  "message": "Booking not found"
}
  • Error (400):
{
  "message": "Invalid payment status"
}

Booking Workflow

1. User Books a Seminar

  • The user creates a booking with the status pending.

2. Admin Reviews the Booking

  • The admin verifies the uploaded proof of payment or other required details.

3. Admin Updates Booking Status

  • The admin updates the paymentStatus to either confirmed or rejected using the new endpoint.
